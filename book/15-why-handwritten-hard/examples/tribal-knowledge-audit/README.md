# tribal-knowledge-audit：审计藏在代码里的手写知识欠账

第 15 章配套示例。扫描 `aishop` 的代码，找出「业务知识已经泄露到代码里、但还没被结构化成知识」
的信号，聚成一份待沉淀清单——把「知识不太够」的模糊焦虑变成可逐条消化的欠账。

## 运行

需要 Node >= 20.11。

```bash
npx tsx src/audit.ts
# 或
npm install && npm start
```

只用 Node 内置模块，无运行时依赖。

## 扫描的三类信号

- **BIZ-RULE 注释**：业务规则只活在一行注释里（如「退款超阈值需人工审核」）。
- **未解释的魔法数字**：`const REVIEW_THRESHOLD = 5000`——为什么是 5000？理由在某人脑子里。
- **业务 TODO/FIXME**：带业务疑问的待办（如「legacy_channel 为什么不能删」）。

## 预期输出

审计 `aishop/src/` 得到 6 条欠账（2 条 BIZ-RULE、2 条魔法数字、2 条业务 TODO/FIXME），
每条都「只以代码信号存在、未结构化、无 owner」。

结论：这份清单就是下一章 promote 要上收的原料——手写业务知识的第一步，
是把只在人脑/注释里的东西，先制造成可寻址的结构化知识文件。

## 说明

这里用简单的正则信号做演示。真实项目可以把它做成 CI 里的一个「知识欠账」指标，
持续跟踪「有多少业务知识还没被结构化」，也可以接更强的信号（如 code review 里反复被问的问题）。
