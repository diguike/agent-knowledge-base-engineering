# extract-review：抽取候选 + 冲突/近重复检测 + 人审队列

第 18 章配套示例。实现「自动抽取 → 近重复检测 → 语义冲突检测 → 人审队列」这条流水线：
从事故复盘、PR 讨论里挖候选知识，检测它和已有知识是否重复或矛盾，带标记交人拍板。

## 运行

需要 Node >= 20.11。

```bash
npx tsx src/main.ts
# 或
npm install && npm start
```

只用 Node 内置模块，无运行时依赖。

## 目录

- `src/detect.ts` —— 已入库知识 + 抽取 + 近重复/语义冲突检测（本地词袋 embedding）
- `src/main.ts` —— 跑流水线：两段原始文本（复盘 + PR 讨论）→ 候选 → 检测 → 人审队列

## 预期输出

已有知识含「退款金额超过 5000 元需人工审核」。三种候选各被正确标记：

- **疑似冲突**：「退款金额超过 3000 元需人工审核」——和已有的 5000 那条高度相似（0.87），但关键数字不同。
- **疑似重复**：「退款金额超过 5000 元需要人工审核」——和已有知识几乎一样（0.73）。
- **新知识，可入库**：「支付回调要加幂等」「大促库存扩容」——和已有知识都不相似。

三种都进人审队列，等人拍板。

> 注：抽取重召回会多挖出一条叙事性噪音句（「本次事故……被薅」），同样落入「新知识可入库」，
> 所以实际队列有 5 条而非 3 条——人审时直接丢弃即可。

## 两个「换成真模型」的插槽

- **抽取**：本例用关键词规则从文本挑候选句（重召回，会带一些噪音，靠人审兜底）；生产用大模型抽取。
- **检测**：本例用本地词袋 embedding 算相似度；生产用真 embedding，语义相近但用词不同也能识别。
  - `DUP_THRESHOLD = 0.5` 是词袋 embedding 下手工试出来的，换真 embedding 后需用一批已知重复/不重复的样本重新标定，不能照抄。
  - 冲突检测本例只做**数字比对**；条件类冲突（如「允许」/「不允许」相反）抓不出来，需生产 LLM 裁决，本例未演示。

近重复 + 语义冲突这两道检测，是 `ai-asset-standards` 现有 CI 还没覆盖的缺口，本例补上其机制。
最终「入库/丢弃/替换旧知识」的判断始终由人做——自动降摩擦，人守质量门。
