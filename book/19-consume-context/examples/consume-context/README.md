# consume-context：上下文预算下的知识组装

第 19 章配套示例。模拟 agent 处理任务时的上下文组装，对比两种策略——「全塞」和
「常驻 L0 + 按需检索」——直观展示「知识建好了，怎么被有效消费」。

## 运行

需要 Node >= 20.11。

```bash
npx tsx src/main.ts
# 或
npm install && npm start
```

只用 Node 内置模块，无运行时依赖。

## 目录

- `src/assemble.ts` —— 知识库（L0 常驻 + L1 领域）+ 相关度计算 + 两种组装策略
- `src/main.ts` —— 对同一任务「写退款接口的风控校验」跑两种策略

## 预期输出

上下文知识预算 50 tokens：

- **全塞**：所有知识都常驻，合计 80 tokens——**超预算**，还塞进「大促库存」「对账」这些和退款任务无关的知识（相关度 0）。
- **常驻 L0 + 按需检索**：L0 约定必放，剩余预算内按相关度检索——只放了 L0 + 最相关的「风控名单」「退款审核」两条，合计 44 tokens，**在预算内**且全相关。

结论：上下文是有限预算，知识和代码、对话争空间；目标是「准」不是「多」。
把召回范围提前圈小（依赖声明/namespace/覆盖度），消费时才能又准又省。

## 说明

token 数这里用显式字段近似（真实按 tokenizer 算），相关度用 2-gram 重合近似（真实用 embedding）。
这两个换成真实现，「常驻 + 按需 + 预算」这套组装策略不变。

L2（仓库本地 `CLAUDE.md`/`AGENTS.md`）和 L0 同属常驻；本例为聚焦「按需检索」的对比，简化为只建模
L0/L1，L2 可视为并入 L0 常驻块。
