# governance-ci：漂移检测 + 健康度看板 + 落地路线图

第 23 章配套示例，也是全书的收尾。三件事：漂移检测（新鲜度）、三位一体健康度看板、落地路线图推荐。

## 运行

需要 Node >= 20.11。

```bash
npx tsx src/main.ts
# 或
npm install && npm start
```

只用 Node 内置模块，无运行时依赖。

## 目录

- `kb/L1/` —— 3 个知识包（1 个正常、1 个超期未复核、1 个已废弃仍在库）
- `src/governance.ts` —— 漂移检测 + 健康度判定 + 路线图推荐
- `src/main.ts` —— 跑三个环节

## 预期输出

1. **漂移检测**：揪出 3 条——`kb-legacy`（status=deprecated 仍在库）、`kb-refund`（last_reviewed 已 177 天 > TTL 90）、`kb-refund`（一致性漂移：知识说阈值 3000，但代码里已是 5000）。
2. **健康度看板**：覆盖度 85% ✓、有效性 90% ✓、新鲜度 3 条漂移 ✗ → 总判定「不健康，CI 标红」。
   （覆盖度、有效性在真实管线里来自第 5、21 章的示例，这里用示例值演示三项汇总。）
3. **落地路线图**：按团队画像推荐停在哪一阶——个人/单仓库→阶段0，跨仓库→阶段1，大规模/跨源→阶段2，跨团队→阶段3。
4. 有漂移导致不健康，脚本以**退出码 1** 结束（可当 CI 门禁）。

结论：把知识库当软件工程——健康度三项（覆盖度+有效性+新鲜度）进 CI、按需求停在合适一阶，够用就别升级。

## 说明

漂移检测的「时间漂移」（last_reviewed+TTL）和「一致性漂移」（知识里的旧阈值 3000 vs 代码现值 5000）本例都真算了；
真实项目里代码值从代码抽取、比对逻辑更复杂，本例用一个 mock 常量演示最小比对。
覆盖度/有效性用示例值，真实管线接第 5/21 章示例的输出。
