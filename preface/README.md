## 一座建歪了的知识库

去年我旁观一个团队给他们的 coding agent 建知识库。第一步，他们就搭了一套向量检索——整个仓库切块、算 embedding、塞进向量库，指望 agent 从此无所不知。两周后，效果比不用还差。

agent 回答"订单状态机怎么流转"时，把三个文件里各半截的代码拼成了一套并不存在的规则，还答得一脸自信。他们没有停下来问方向对不对，而是继续调 chunk 大小、换 embedding、加 rerank。又一个月过去，基础设施养起来了，agent 该错还是错。

问题从一开始就不在参数上。那个"订单状态机"的边界清清楚楚摆在几个文件里，grep 一个符号就能定位，根本用不上为"不知道答案在哪"而设计的重机制。

更要命的是另一类错：agent 一次次把退款金额算错。因为"退款超过 5000 要人工审核"这条规则不在任何代码文件里——它在财务同事的脑子里，在两年前一次事故复盘的口头结论里。再强的模型、再贵的向量库，也装不下一个从没被写下来的东西。

这样的团队我见了太多。这本书就是写给他们，也写给正准备走上同一条路的你。

## 为什么写这本书

市面上讲"给 agent 建知识库"的内容，几乎都在教同一件事：怎么搭一个 RAG。但 RAG 只是知识库可能用到的一种检索机制，不是知识库本身。

真正决定一个知识库好不好用的，是它背后的工程：知识怎么组织、怎么分发、怎么让整个团队持续共建、怎么度量它到底有没有用。这些工程的部分，恰恰没有人系统地讲。

尤其是共建。代码衍生的知识（某个函数怎么调、某个 API 什么参数）能被工具自动知识库化，是因为它本就是仓库里可寻址的结构化文件，抽取的第一步是免费的。

但企业真正值钱的知识——为什么退款要先过风控、这个字段为什么不能删、大促库存为什么扩容三倍——散落在人的记忆和聊天记录里，没有那个可以自动抽取的仓库。**把这些手写业务知识沉淀下来，是最难、也最值钱的一块**，也是让 agent 从会写通用代码变成懂你们业务的护城河。这本书用最重的篇幅去攻它。

一句话概括本书的立场：**把 Agent 知识库当软件工程来做**——分层组织、包化分发、CI 集成、共建治理，每一步都给出开源、可自托管、能跑通的方案。

## 这本书写给谁

我预设的读者有两类。一类是正在转型的传统工程师：前端、后端、全栈背景，想往 Agent 基础设施深处走。另一类是已有初步经验的 AI/Agent 工程师，需要在自己的真实场景下系统性地建一座知识库，而不是拼几个 demo。

无论哪一类，你的处境大概相似：手上有一个真实的代码仓库和一堆散落的业务知识，agent 用起来时灵时不灵，你想把它治理成一套可靠的基础设施。书里的示例以 Node + TypeScript 为主，前端全栈背景读起来会很顺。

有两类人这本书帮不上忙，先说清楚免得浪费你时间。一类是只想一键接入某个 SaaS、完全不关心知识怎么组织和治理的人，你需要的是那个产品的说明书。另一类是想深入 RAG 算法、embedding 数学推导的人，这本书是工程落地导向，遇到算法边界只讲工程选型，不展开推导。

## 这本书覆盖什么，不覆盖什么

全书围绕一个虚构的电商后端 `aishop` 展开——它有订单、库存、支付、退款、风控、对账几个模块，既有代码衍生的知识，也有只在人脑里的业务规则。你要给它建的那座知识库，我们叫它 `aishop-kb`。

它从一个空目录起步，沿一条能力阶梯一级级长大：一个 Markdown 文件夹 → 分层知识包 → MCP 知识服务 → 多包声明式分发。读到第 22 章，你手里会有一座完整、可自托管、能自检的 aishop-kb，以及一套随书逐章长出来的 CLI：coverage 扫覆盖度、serve 起知识 MCP 服务、promote 把本地随手记上收成共享包、check 做质量门禁、extract 从事故复盘抽候选、eval 跑评测、drift 检测漂移。

就像《自己动手写 AI Agent》带你亲手写出 Ling、评测那本书带你建出 EvalKit，这本书带你从一个空目录建出 aishop-kb。每一章的代码都在这个仓库上继续生长，你跟着敲，最后手里是一个真东西，不是一堆截图。

不覆盖的是：RAG 检索算法的数学细节、向量数据库的性能调优、以及任何闭源 SaaS 的点按操作。我也不装有银弹——书里反复强调"够用就别升级"，很多团队停在能力阶梯第一级就够了，不必为用不上的规模买单。

## 怎么读这本书

我给四类读者四条路线，每条都标了理由，别囫囵吞枣从头读到尾。

- **只想快点用起来**：读第一部分建立心智，直接跳到第三部分把最小的 aishop-kb（一个文件夹 + 索引 + 确定性导航）搭起来。够用就停在那，别急着往上爬。
- **想系统掌握**：从头顺着读，跟着 aishop-kb 一章章长大，每部分配套的 `examples/` 都跑一遍。读完你会有一套完整心智和一个可复用的代码骨架。
- **只为解决共建难题**：直奔第五部分。手写业务知识怎么低摩擦沉淀、怎么共建、怎么从事故复盘里自动抽取，是全书最难也最值得单独读的一块。
- **只关心怎么证明知识库有用**：直奔第二部分（覆盖度前置）和第六部分（度量与治理）。这两部分单独成线，讲怎么量化该有的知识有没有、有了准不准、会不会过期，不需要先啃分层组织和 MCP 分发。

这本书是我和 Claude 协作完成的。如果你在阅读或运行示例时发现错误，欢迎反馈——知识库最怕过期而无人更正，一本讲知识库治理的书，更不该自己腐化。
